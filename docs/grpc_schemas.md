# xbin gRPC Schema Documentation

This document describes the gRPC protocol used by the **xbin** orchestrator and its fleet of analysis workers.

## Service: `OrchestratorService`

The central service that coordinates the analysis blackboard. Workers act as clients to this service.

### 1. `RegisterWorker`
Used by a worker to join the xbin fleet upon startup.

- **Request (`RegisterRequest`)**:
    - `worker_id` (string): A unique identifier for the specific container instance (usually generated by the SDK).
    - `backend_name` (string): The name of the tool (e.g., `angr_cfg`).
    - `analysis_type` (string): The blackboard category this tool contributes to (e.g., `cfg_generation`).
- **Response (`RegisterResponse`)**:
    - `success` (bool): True if the orchestrator acknowledged the registration.

### 2. `Heartbeat`
A persistent signal sent by workers every 5 seconds to maintain their "READY" status on the dashboard.

- **Request (`HeartbeatRequest`)**:
    - `worker_id` (string): The ID provided during registration.
    - `status_message` (string): Current activity status (e.g., "Online & Ready", "Analyzing a.out").
- **Response (`HeartbeatResponse`)**:
    - `acknowledged` (bool): Acknowledgment from the orchestrator.

### 3. `PostResult`
The primary method for contributing analysis to the blackboard. This method triggers a global broadcast to all other tools.

- **Request (`PostResultRequest`)**:
    - `analysis_type` (string): The category (blackboard) for this result (e.g., `symbol_matching`).
    - `item_key` (string): The unique identifier for the item being analyzed (e.g., a function address `0x401000` or a filename `ls.bin`).
    - `result_data` (string): A **JSON-encoded string** containing the actual analysis payload.
    - `confidence` (float): The tool's certainty (0.0 to 1.0).
    - `backend_name` (string): The tool's identity.
- **Response (`PostResultResponse`)**:
    - `accepted` (bool): True if the hypothesis was recorded.
    - `current_status` (string): The new resolution status of the item (e.g., `RESOLVED`, `CONFLICTED`).

## Analysis Category Data Standards (`result_data`)

To enable competition and consensus, tools in the same category must follow shared JSON schemas for the `result_data` field.

### Category: `symbol_matching`
Identifies the name of a function at a specific address.
- **`item_key`**: Function start address (hex string).
- **`result_data`**: A plain string representing the symbol name.
- **Example**: `"main"`

### Category: `cfg_generation`
Recover the Control Flow Graph (CFG) of a binary.
- **`item_key`**: Filename of the binary.
- **`result_data`**: A dictionary containing `nodes` and `edges`.
- **Example JSON**:
  ```json
  {
    "nodes": [
      {"id": "0x401000", "label": "main"},
      {"id": "0x401020", "label": "0x401020"}
    ],
    "edges": [
      {"source": "0x401000", "target": "0x401020"}
    ]
  }
  ```

### Category: `function_boundary`
Identifies the start and end of functions.
- **`item_key`**: Function start address (hex string).
- **`result_data`**: A dictionary with boundary metadata.
- **Example JSON**:
  ```json
  {
    "end": "0x401045",
    "size": 69,
    "name_hint": "main"
  }
  ```

## Feedback & Evolution
The **xbin** protocol is designed to be highly extensible. If you are building a new type of analysis (e.g., Taint Analysis or Type Recovery) and have suggestions for standardizing the `result_data` schema for your domain, we highly value your input. 

Please reach out or open a PR to update this specification!
